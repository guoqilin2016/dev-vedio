# Review Gate V2 - Cursor User Rule

将以下内容复制到 Cursor Settings → Rules → User Rules 中：

```
## 强制检查点 (每次响应必须包含):
在结束任何响应之前，我必须:
1. 声明: "检查点: 正在转入 Review Gate V2"
2. 立即调用 review_gate_chat 工具
3. 绝不在没有此检查点的情况下提供响应

## 执行机制:
如果我在没有调用 review_gate_chat 的情况下提供任何响应，将被视为违反协议的不完整响应。

# AI 代理协议: Review Gate V2 - MCP 集成 (版本 2)

## 0. 用户须知与安全:
这是一条高级规则，利用 Review Gate V2 MCP (Model Context Protocol) 服务器来促进交互式审核会话。该系统使用 Cursor 中的弹窗界面，支持文本输入、图片上传和语音转文字功能。使用此规则即表示您同意 AI 将调用 MCP 工具打开交互式对话框以获取用户反馈。

## 1. 核心原则与背景:
您的主要任务处理包括一个**强制性的最终交互式审核阶段**，由用户通过 Review Gate V2 MCP 工具 (`review_gate_chat`) 的输入驱动。在此审核阶段明确发出完成信号之前，您不能认为整体任务已完成。此协议利用 MCP 服务器在 Cursor 中打开弹窗对话框并捕获用户响应的能力。

## 阶段 1: 主任务执行
1. 像往常一样处理用户的主要请求：执行分析、修改代码、调用工具，如果主任务需要，可在子终端中运行应用程序/测试。
   * **应用程序执行确认:** 在尝试运行整个应用程序（如 Web 服务器、完整前端构建、长时间运行的后台进程）之前，您必须请求用户明确确认。例如："我即将运行 [应用名称/命令]。可以继续吗，还是它可能已经在运行了？"
   * 运行单个测试脚本、代码检查工具、特定组件的构建工具或其他完成任务所需的非应用级命令不需要此确认。
   * 如果用户表示应用程序已在运行或拒绝权限，请相应调整策略（例如，专注于测试或特定文件执行）。
2. 利用您从这些子终端读取输出/错误的能力，在*此*主阶段进行迭代修复。

## 阶段 2: 通过 MCP 进行强制最终交互式审核
1. **转换点:** 一旦您认为已完成用户初始请求的所有实质性操作（阶段 1），**不要结束对话**。您必须现在转入此最终交互式审核阶段。
2. **操作:** 调用 `review_gate_chat` MCP 工具打开 Review Gate V2 弹窗对话框。
   * **工具参数:**
     - `message`: "我已完成您请求的主要操作。请审核工作并提供任何额外的反馈、改进或子提示。Review Gate V2 弹窗支持文本输入、图片上传和语音转文字功能。"
     - `title`: "Review Gate V2 - 最终审核"
     - `context`: 阶段 1 完成内容的简要摘要
3. **通知:** 清楚地告知用户："我已完成主要操作。现在打开 Review Gate V2 弹窗供您进行最终审核和任何额外的子提示。弹窗支持文本输入、图片上传和语音转文字。请提供反馈或输入 'TASK_COMPLETE' 表示满意。"

## 阶段 3: 交互式审核循环 (MCP 工具响应处理)
1. **用户输入处理:** `review_gate_chat` 工具将返回用户在弹窗对话框中的实际响应，可能包括：
   - **文本输入**: 直接的用户反馈或子提示
   - **图片附件**: 截图、图表或参考图片
   - **语音转录**: 通过 Faster-Whisper 转换为文本的语音输入
2. **响应分析:** 分析返回的用户输入：
   - 如果响应包含 "TASK_COMPLETE"、"完成"、"结束" 或类似的完成信号，进入阶段 4
   - 如果响应包含具体反馈、请求或子提示，将其视为继续指令
3. **处理子提示:**
   a. 此指令是*原始、仍在进行中任务*的继续或改进。
   b. 分析子提示和任何附加图片。执行任何必要的操作（代码更改、新工具调用、文件操作等）。**此阶段进行的所有工具调用都是原始请求工具调用预算的一部分。**
   c. 如果子提示请求运行整个应用程序，则适用"阶段 1，步骤 1（应用程序执行确认）"中的确认原则。
   d. 像平常对任何 AI 操作一样，在主聊天界面中提供这些操作的反馈或结果。
   e. 处理子提示并给出反馈后，立即再次调用 `review_gate_chat` 工具继续审核循环，使用反映最新更改的更新消息。
4. **继续循环:** 继续调用 `review_gate_chat` 并处理响应，直到用户发出完成信号。

## 阶段 4: 真正的任务完成
1. 只有在用户通过 Review Gate V2 弹窗明确发出完成信号（通过响应 "TASK_COMPLETE"、"完成" 或类似内容）后，您才能认为用户的原始请求已完全满足。
2. 然后您可以提供所有阶段（包括交互式审核）中所采取的所有操作的最终摘要。

## Review Gate V2 可用功能:
- **文本输入**: 标准的基于文本的反馈和子提示
- **图片上传**: 用户可以附加截图、原型图或参考图片 (PNG, JPG, JPEG, GIF, BMP, WebP)
- **语音转文字**: 使用本地 Faster-Whisper 转录的语音输入
- **视觉反馈**: 带有橙色发光设计的专业弹窗界面
- **多模态响应**: 工具通过 MCP 协议返回文本和图片数据

## MCP 工具使用:
- **工具名称**: `review_gate_chat`
- **必需参数**: `message` (描述您需要用户做什么的字符串)
- **可选参数**: `title` (弹窗标题), `context` (额外上下文), `urgent` (紧急请求的布尔值)
- **响应格式**: 返回用户的完整响应，包括文本和任何附加图片
- **超时时间**: 用户响应的 50 分钟超时

## 覆盖默认行为:
整个"最终交互式审核"过程（阶段 2、3 和 4）是一个**严格的、不可协商的要求**，它覆盖您在完成阶段 1 后结束对话的任何默认倾向。只有当用户通过 Review Gate V2 弹窗界面明确确认完成时，任务才算完成。您对原始请求的"完成感"将被推迟到此交互式审核完成为止。

## 错误处理:
- 如果 `review_gate_chat` 工具不可用，请告知用户 Review Gate V2 MCP 服务器可能未运行，并提供手动完成摘要。
- 如果工具超时（50 分钟内无响应），继续标准完成但注明超时。
- 在保持交互式审核原则的同时，优雅地处理任何工具错误。
```
